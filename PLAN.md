# PLAN.md

## Task

Реализовать P0-стабилизацию BugZero по утверждённому ТЗ: подготовить безопасный переход SQLite -> PostgreSQL и устранить критические риски доступа, целостности и качества.


## Context

ТЗ и технический аудит завершены. Начинаем с подготовительного среза: фиксируем исполнимый тестовый контур и контракт PostgreSQL, не меняя рабочую SQLite БД и не создавая внешнюю инфраструктуру без решения владельца.


## Approach

1. Утвердить PostgreSQL-контур и параметры cutover, необходимые для миграции данных.
2. Создать тесты для первого безопасного P0-среза и подтвердить их RED-состояние.
3. Реализовать минимальный срез, довести тесты до GREEN и зафиксировать контракты deployment/миграций.
4. Продолжить P0 по отдельным малым изменениям: RBAC, доменная схема, миграция данных, storage и UI-моки.


## Milestones

- [x] **M1:** Анализ и ТЗ — verify: исходный код, SQLite, deployment scripts, тесты и история PostgreSQL проверены; ТЗ дополнено.
- [x] **M2:** Подготовка первого P0-среза — verify: выбран собственный PostgreSQL-сервер; RED-тесты для runtime-контракта, install и initial migration подтверждены.
- [x] **M3:** Первый P0-срез — verify: Prisma переведена на PostgreSQL, добавлены базовая миграция и `.env.example`, штатные scripts используют `migrate deploy`; 6 тестов, `prisma validate` и build с PostgreSQL URL проходят.
- [x] **M4:** Миграция и стабилизация P0 — verify: локальные PostgreSQL `bugtrack_dev` и `bugtrack_test` созданы, получают одну Prisma migration; тестовые SQLite-данные по решению владельца не переносятся. RBAC, доменная схема, private storage, audit history, soft delete, CI и error UX реализованы.

Остаются настройка колонок и массовые операции, WIP Kanban, полная карточка задачи и responsive/design-system срезы.

## Scope boundaries

In scope:

- Подготовка тестового и миграционного контуров P0.
- Реализация P0 небольшими изолированными срезами после RED-тестов.

Out of scope:

- Cutover рабочей базы и деплой без утверждённого PostgreSQL provider, доступа и maintenance window.
- Установка новых зависимостей без отдельного согласования.

## Risks

- В рабочем дереве есть пользовательские незакоммиченные файлы и изменённая SQLite БД; они не изменяются.
- В текущей реализации уже есть несовместимые трактовки «проект/команда» и глобальных статусов; миграция требует резервной копии и отдельного плана.
- Исторические SQLite-данные признаны тестовыми и не переносятся. PostgreSQL используется как чистая целевая БД; staging/cutover остаются отдельной операционной задачей.
