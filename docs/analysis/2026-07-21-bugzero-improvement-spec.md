# BugZero: аудит и техническое задание на улучшения

Дата: 21.07.2026
Статус: к согласованию
Целевая аудитория: владелец продукта, техлид, frontend/backend-разработчики, QA

## 1. Резюме решения

BugZero уже имеет хороший фундамент: аутентификацию, задачи, комментарии, вложения, уведомления, список, Kanban и административные экраны. Однако продукт сейчас находится между прототипом и production MVP. Главная проблема не в нехватке экранов, а в несогласованности доменной модели и реализации: «проект» в интерфейсе фактически является командой, часть маршрутов показывает тестовые данные, статусы одновременно задуманы как командные и реализованы как глобальные, а Server Actions не проверяют членство в команде.

Рекомендуемая последовательность:

1. **P0. Сделать существующий MVP корректным и безопасным.** Переход SQLite -> PostgreSQL во всех окружениях, единая модель Workspace -> Team -> Project -> Cycle -> Issue, проверка прав, миграция статусов, устранение моков, валидация и типизация.
2. **P1. Сделать его быстрым и удобным для ежедневной работы.** Полноценные список и Kanban, быстрые действия, командная палитра, UX состояний, адаптивность.
3. **P2. Добавить управляемость и интеграции.** Циклы, аналитика, GitHub/GitLab, Slack, SLA и наблюдаемость.

Целевая ценность: инженерная команда из 5-50 человек должна создать баг, назначить исполнителя, обсудить его, сменить статус и увидеть актуальный прогресс без переключения в сторонний инструмент.

## 3. Продуктовые решения, которые надо принять и реализовать

### 3.1. Каноническая модель предметной области

Необходимо зафиксировать терминологию и не использовать слова «команда» и «проект» как взаимозаменяемые.

```text
Workspace
  └─ Team (люди, роли, свой workflow)
       └─ Project (цель/инициатива, даты, прогресс)
            └─ Cycle (необязательно; временной интервал команды)
                 └─ Issue (bug / task / improvement)
```

- `Issue.teamId` обязателен.
- `Issue.projectId` и `Issue.cycleId` необязательны.
- Добавить обязательный `Project.teamId`; проект нельзя связывать с задачами другой команды.
- `WorkflowStatus` принадлежит `Team`, а не всему workspace. Добавить `@@unique([teamId, name])`.
- Добавить типы вместо строк: `UserRole`, `TeamRole`, `IssueType`, `Priority`, `IssueStatusType`.
- Номер задачи генерировать транзакционно через счётчик команды или sequence. Формат: `{TEAM_KEY}-{sequence}`, например `BUG-123`. Нельзя использовать `count + 1`.

### 3.2. Роли и доступ

Реализовать RBAC на сервере, а не только скрытие кнопок:

| Операция | GUEST | MEMBER | TEAM_ADMIN | WORKSPACE_ADMIN |
|---|---:|---:|---:|---:|
| Читать задачи команды | назначенные/созданные | да | да | да |
| Создать задачу | да | да | да | да |
| Изменить чужую задачу | нет | да | да | да |
| Управлять участниками, проектами и workflow команды | нет | нет | да | да |
| Управлять workspace | нет | нет | нет | да |

Требования:

- Создать `requireTeamRole(teamId, minimumRole)` и `requireIssueAccess(issueId, action)`; вызывать их во всех Server Actions и API routes до чтения/мутации данных.
- Пользователь видит в фильтрах и поиске только команды, где он участник.
- Запись audit history должна покрывать создание, изменение title/description/priority/status/assignee/project/cycle/labels, удаление и скачивание вложений.
- Удаление задачи: soft delete (`deletedAt`, `deletedById`) и доступно только администратору команды; восстановление в течение 30 дней.

### 3.3. Граница авторизации: NextAuth, Prisma и PostgreSQL RLS

Текущая система использует NextAuth и один серверный Prisma connection pool. Пользователь приложения не является отдельным PostgreSQL-пользователем, поэтому простая RLS-политика через `current_user` не различит участников BugZero. Нельзя считать включение RLS заменой проверок в Server Actions.

Для P0 принять следующую архитектуру:

- Основная авторизация выполняется на сервере функциями `requireTeamRole` и `requireIssueAccess` внутри каждой операции чтения и записи.
- PostgreSQL недоступен из браузера и публичной сети; приложение подключается ролью с минимальными правами, миграции — отдельной ролью.
- Если требуется дополнительная PostgreSQL RLS, идентификатор пользователя и команды передаётся через request-scoped transaction context (`SET LOCAL`) и проверяется политиками в той же транзакции. Connection pool нельзя использовать без очистки контекста.
- Переход с NextAuth на Supabase Auth и прямой доступ клиента к таблицам — отдельный ADR и отдельная фаза; в миграцию БД P0 он не входит.

Критерий приёмки: автоматический тест доказывает, что Server Action не позволяет прочитать/изменить чужую команду. Если включена RLS, отдельный SQL integration test подтверждает ту же изоляцию для роли приложения без request context; если RLS не включена, подключение этой роли должно быть доступно только backend-сервису в приватной сети.

## 4. P0: обязательная стабилизация MVP

### 4.1. Переход SQLite -> PostgreSQL

Текущая SQLite БД (`prisma/dev.db`) является только источником одноразовой миграции. После cutover PostgreSQL 16+ должен использоваться одинаково в local, test, staging и production. Один Prisma schema не должен поддерживать два разных provider: это создаёт расхождения типов, миграций, поиска и транзакционного поведения.

Рекомендуемый production-вариант — managed PostgreSQL (Supabase, Neon или другой совместимый провайдер). При выборе Supabase использовать PostgreSQL и private Storage; перенос аутентификации на Supabase Auth не является обязательным. Локальная разработка работает с PostgreSQL в Docker либо с отдельной development database/branch.

Разработчик должен:

1. Завести независимые PostgreSQL базы для `local/test`, `staging` и `production`; секреты подключений хранить только в переменных окружения.
2. Заменить в `prisma/schema.prisma` datasource provider на `postgresql`; не поддерживать параллельную SQLite-схему и не менять production schema вручную.
3. Добавить безопасный `.env.example` без секретов. Для managed pooler использовать `DATABASE_URL` приложения и отдельный `DIRECT_URL`/роль для миграций, если это требуется выбранным провайдером.
4. Создать чистую базовую Prisma migration и проверить `prisma validate`, `prisma generate` и `prisma migrate deploy` в CI на пустой PostgreSQL БД.
5. Подготовить одноразовый idempotent-скрипт переноса: SQLite export -> проверка -> PostgreSQL import. Порядок импорта: users, teams, team members, statuses, projects/cycles, issues, labels, связи, comments/history/notifications, metadata вложений.
6. Сохранять текущие числовые `Issue.id`; после импорта явно синхронизировать PostgreSQL sequence (`setval`) с максимальным ID, иначе первая новая задача может конфликтовать с импортированной.
7. До переключения выполнить backup SQLite и PostgreSQL, dry run на staging, сверку количества записей, внешних ключей и контрольных сумм бизнес-идентификаторов (`readableId`, нормализованный email, team key).
8. Провести cutover в maintenance/read-only окне: остановить записи в SQLite, сделать финальный export, импорт и smoke test, затем открыть запись в PostgreSQL. Rollback на SQLite допустим только до открытия новых записей; dual-write не реализовывать.
9. Настроить connection pooling, SSL, timeout запросов, ежедневные backup/PITR и alert на исчерпание подключений/ошибки БД.
10. Исторический commit `824712e` использовать только как справочный черновик: он менял provider и создавал пустую схему, но не переносил данные и был откатан commit-ом `9af3ada`.

Обязательные изменения deployment-контракта:

- заменить `npx prisma db push` на `npx prisma migrate deploy` в `deploy.sh`, `install.sh`, BAT/SH-скриптах production-профиля; локальная разработка использует `prisma migrate dev`, а не отдельную SQLite-схему;
- заменить `test/sqlite-deployment.test.mjs` тестом PostgreSQL-конфигурации и обновить `test/deploy-script.test.mjs`;
- добавить `prisma/migrations/migration_lock.toml`, базовую migration и проверку отсутствия schema drift;
- унифицировать порт через `APP_PORT`: сейчас scripts используют 3000, 3002 и 3008;
- выполнять DB preflight до миграции, затем build -> `pm2 reload` -> readiness check; при неуспешном readiness вернуть предыдущую версию приложения. Миграции P0 должны следовать expand/contract и оставаться совместимыми с предыдущей версией на время rollback.

Критерии приёмки:

- Все local/test, staging и production migrations применяются через Prisma без ручного SQL hotfix.
- После dry run число сущностей и все `readableId` совпадают с источником; ссылки на существующие задачи открываются.
- `npm run build`, интеграционные тесты и E2E smoke test работают с PostgreSQL.
- Ни одно runtime-окружение не использует `file:./dev.db`; SQLite snapshot хранится только как защищённый архив миграции с ограниченным сроком хранения.
- После импорта создаётся новая задача с корректным следующим ID, поиск одинаково работает с латиницей и кириллицей.

### 4.2. Данные и доменная миграция

Разработчик должен:

1. Сделать backup рабочей БД и подготовить обратимую Prisma migration.
2. Внести модель из раздела 3.1, мигрировать текущие `Team`/`WorkflowStatus`/`Issue` без потери задач, комментариев и вложений.
3. Убрать глобальные статусы. Для каждой команды создать начальный workflow: `Backlog`, `Todo`, `In Progress`, `In Review`, `Done`, `Canceled`; тип статуса определяет смысл, имя можно менять.
4. Перенести все операции с проектами на `Project`; текущий административный экран команд переименовать и использовать отдельно.
5. Реализовать реальные страницы `/teams/[id]` и `/projects/[id]` с данными БД. Запрещено оставлять mock arrays, `console.log` или «would go here» в production code.
6. Перед включением RBAC выполнить backfill `TeamMember`. В текущей БД записей членства нет: workspace-admin получает административное членство по явному правилу, остальные связи формируются из согласованной матрицы, а не угадываются по авторству/назначению задачи.
7. Мигрировать смешанные статусы: текущая база содержит global и team-specific записи. Для каждой команды создать собственный набор, переназначить issues, затем сделать `WorkflowStatus.teamId` обязательным.
8. Отключить и удалить из эксплуатационного контура `migrate_to_global_statuses.js`, debug-скрипты и неидемпотентное создание статусов в seed. Seed тестовой среды должен быть повторяемым и не использовать известные пароли.
9. Добавить ограничения целостности: уникальные `[teamId, name]` и `[teamId, position]` для статусов; `Project.teamId`; проверку принадлежности status/project/cycle/label той же команде, что и issue.
10. Перенести физические файлы из `public/uploads` в private object storage: вычислить checksum, проверить наличие каждого файла, создать закрытый object key, обновить metadata и сформировать отчёт о потерянных/повреждённых вложениях. Старые публичные URL отключить только после проверки новых signed URL.

Критерии приёмки:

- Созданная задача принадлежит ровно одной команде и отображается только её участникам.
- Новый статус одной команды не появляется у другой.
- Параллельное создание 20 задач одной команды не даёт дублирующихся `readableId`.
- Существующие данные после миграции доступны по прежним ссылкам.
- До включения ограничений сформирован migration report: строки без владельца/команды, исправленные статусы, созданные memberships и отклонённые связи.

### 4.3. Безопасность и валидация

- Для каждой Server Action создать Zod schema и возвращать единый результат: `{ success, data?, fieldErrors?, errorCode? }`.
- Ограничения задачи: title 3-180 символов, описание до 20 000 символов, допустимые enum-значения, assignee только из той же команды.
- Разрешать загружать только whitelist MIME types (изображения, PDF, текстовые логи), проверять размер до чтения файла, лимит 10 файлов / 25 MB на задачу и 10 MB на файл. Значения согласовать с владельцем продукта, но тесты должны быть параметризованы.
- Хранить вложения в private object storage; скачивание осуществлять через авторизованный signed URL. Не хранить пользовательские файлы в `public/uploads` в production.
- Добавить rate limit на создание задач, комментарии, загрузку и регистрацию; журналировать отказ доступа без передачи чувствительных данных пользователю.
- Удалить debug output сессии и заменить `alert` на обработанные UI-уведомления.
- Закрыть публичную регистрацию либо перевести её в invitation-only режим. Нормализовать email (`trim + lowercase`), ввести проверку сложности пароля, rate limit входа/регистрации/сброса и отзыв активных сессий после смены пароля.
- Запретить известные пароли из seed и fallback-пароль `123456`; временный пароль должен быть случайным, одноразовым и требовать смены при первом входе.
- Проверять роль внутри каждой admin Server Action. Защита `app/admin/layout.tsx` не защищает прямой вызов `actions/users.ts`, `actions/projects.ts` или `actions/workflow.ts`.

Критерии приёмки:

- Участник команды A получает `FORBIDDEN` при чтении/изменении issue команды B даже при ручной подстановке ID.
- Неверные данные не попадают в БД; форма показывает ошибку у конкретного поля.
- Файл сверх лимита или недопустимого типа не сохраняется.
- Неавторизованный прямой вызов каждой admin action возвращает `FORBIDDEN` и не изменяет БД.

### 4.4. Корректность базовых сценариев

- Исправить `CreateIssueModal`: hooks вызываются всегда в одном порядке, список статусов загружается для выбранной команды, плюс в Kanban предзаполняет выбранный статус.
- При перемещении карточки показывать optimistic update, блокировку повторного drag на время запроса, rollback при ошибке и toast с понятным текстом.
- Внести в `updateIssue` проверку всех изменяемых полей, обновлять history для каждого изменённого поля и не создавать дублирующее уведомление.
- Реализовать labels в карточке задачи: добавление, удаление, фильтр по labels. Методы `actions/labels.ts` должны быть защищены team access.
- Упростить sidebar: «Inbox» должен показывать только назначенные текущему пользователю открытые задачи; «All issues» — отдельный URL/режим без дублирования ссылки `/`.

## 5. P1: ежедневная продуктивность и UX

### 5.1. Список задач

Сделать список главным рабочим экраном:

- Колонки: ID, type, title, priority, status, assignee, labels, project, cycle, updatedAt. Пользователь может скрыть/показать колонки и менять их ширину; настройка хранится в профиле.
- Фильтры: команда, проект, цикл, статус, исполнитель, приоритет, type, label, «только мои», «только без исполнителя», дата обновления. Фильтры синхронизируются с URL, чтобы ссылкой можно было поделиться.
- Сортировка по каждому полю, cursor pagination, состояние «0 результатов» с кнопкой сброса фильтров.
- Shift+click и checkbox для массового изменения статуса, исполнителя, priority и labels. Массовая операция должна показать число выбранных задач и подтверждение для destructive действия.

### 5.2. Kanban

- Горизонтальная прокрутка без ломки карточек; фиксировать заголовки колонок при вертикальном скролле.
- Добавить WIP limit на статус, индикатор превышения и предупреждение перед переносом.
- Группировка переключателем: без группировки / по исполнителю / по priority.
- Карточка: цветовой маркер priority, type, labels, assignee avatar, возраст задачи и признаки комментариев/вложений. Не использовать один только цвет как носитель смысла.
- Добавить понятные `aria-label` / tooltip ко всем кнопкам-иконкам и keyboard drag или альтернативное меню «Переместить в…».

### 5.3. Карточка задачи

- Редактирование title, description (Markdown с preview), priority, type, status, assignee, project, cycle и labels без отдельного неполного modal.
- Шаблон создания bug: **шаги воспроизведения**, **ожидаемый результат**, **фактический результат**, **окружение**, **версия**, **вложения**. Для type=bug первые три поля обязательны при переводе в `Todo`.
- Связи задач: blocks / blocked by / duplicates / relates to; родительская задача и подзадачи с индикатором прогресса.
- Activity timeline: человекочитаемые события, комментарии, ссылки на прикреплённые файлы. Сортировка новых снизу, быстрый `Cmd/Ctrl+Enter` для отправки комментария.

### 5.4. Командная палитра и горячие клавиши

Существующая палитра уже открывается по `Cmd/Ctrl+K`; нужно довести её до рабочего инструмента:

- Реализовать предложения, которые сейчас только отображаются: Create Issue, Search, Board, Settings.
- Команды: `c` — новая задача, `g затем i` — список, `g затем b` — доска, `g затем p` — проекты, `/` — поиск, `Esc` — закрыть модальное окно.
- Fuzzy search по ID, title, команде и проекту; debounce 150-250 ms, limit 20, подсветка совпадений.
- Контекстные команды на странице issue: сменить status, назначить, задать priority, скопировать branch name `{teamKey}/{readableId}-{slug}`.

## 6. Визуальное ТЗ

Цель — сохранить спокойный «инженерный» стиль текущей Stone & Ink темы, но сделать интерфейс плотнее, информативнее и доступнее.

### 6.1. Единая дизайн-система

- Вынести inline styles в токены и компоненты: `Button`, `IconButton`, `Input`, `Select`, `Badge`, `Toast`, `Modal`, `EmptyState`, `Skeleton`, `DataTable`.
- Использовать шкалу 4 / 8 / 12 / 16 / 24 / 32 / 48 px; radius 6 px для контролов, 10-12 px для карточек; не смешивать значения без причины.
- Типографика: 12 px metadata, 14 px основной UI-текст, 16 px body, 20/24 px заголовки. Line-height: 1.5 для текста, 1.2 для заголовков.
- Семантические токены: `--success`, `--warning`, `--danger`, `--info`, `--focus`; контраст текста минимум 4.5:1, интерактивных границ/иконок минимум 3:1.
- Добавить dark theme, системный режим и ручной переключатель. Предпочтение сохранять в профиле.

### 6.2. Компоновка

- Desktop (>=1280): sidebar 240 px, контент до 1440 px, max-width для детали issue 1280 px.
- Tablet (768-1279): sidebar сворачивается до иконок/выдвижной панели; фильтры переносятся в одну строку с кнопкой «Фильтры (n)».
- Mobile (<768): sidebar превращается в drawer, padding 16 px, header переносится, основной CTA остаётся видимым; Board становится горизонтально скроллируемой областью; детализация задачи — один столбец, свойства раскрываются после описания.
- Добавить skip link, видимый focus ring, логический порядок tab, подписи к icon-only controls, `aria-live` для toast и status updates.

### 6.3. Состояния интерфейса

- Вместо «Loading...» показывать skeleton соответствующей формы: строки списка, карточки доски, detail sidebar.
- Для пустых состояний: причина + один CTA. Пример: «В этом цикле пока нет задач» + «Создать задачу».
- Для ошибки: название проблемы, следующее действие («Повторить», «Сбросить фильтры», «Вернуться к списку»); без технических трассировок.
- Все сохранения отображаются toast: success, error, undo для обратимых действий. Использовать не блокирующий `alert` только для системно критичных сценариев нельзя.

Критерии приёмки UI:

- Рабочие страницы проходят keyboard-only smoke test: открыть палитру, создать задачу, изменить status и закрыть toast без мыши.
- При ширине 375 px нет горизонтального переполнения body; все критичные действия доступны.
- Lighthouse accessibility >= 90 на списке, доске и карточке задачи; визуальная проверка контраста обязательна.

## 7. P2: управляемость, интеграции и масштабирование

### 7.1. Cycles и аналитика

- Цикл: name, startsAt, endsAt, goal, teamId, статус draft/active/completed. Одна активная итерация на команду по умолчанию; задача может быть перенесена между циклами с историей.
- Dashboard команды: created/resolved trend, throughput, cycle completion, WIP, aging, lead time, MTTR. Каждая метрика должна иметь определение, фильтр периода и ссылку на исходную выборку задач.
- Burn-down только для выбранного цикла; нельзя рисовать «красивую» метрику при отсутствующих estimate points. Если estimates не согласованы, использовать количество задач и явно подписать методику.
- SLA policy для bug: priority -> время до first response / resolution, бизнес-календарь и timezone команды; предупреждения до breach и отчёт по breach.

### 7.2. Интеграции

- GitHub/GitLab OAuth app: привязка репозитория к команде, webhook с signature verification и idempotency key. Парсить `Fixes BUG-123`, `Closes BUG-123`, ссылку на PR/commit; переходить в `In Review` или `Done` только по настраиваемому правилу.
- Slack/Teams: сообщение в выбранный канал только при `URGENT` и при SLA breach; настройки на уровне команды, возможность отключить тип события.
- Внешние вызовы должны иметь retry, DLQ/журнал delivery и не блокировать сохранение задачи.

### 7.3. Нефункциональные требования

- Цели: LCP < 1.5 s на p75, UI feedback < 100 ms, p95 read API < 250 ms, p95 mutation < 500 ms без файла. Метрики собирать, а не декларировать.
- PostgreSQL 16+ во всех runtime-окружениях, миграции Prisma в CI, managed object storage, backup/PITR, мониторинг ошибок и audit logs. SQLite допустима только как источник/архив завершённой миграции.
- Полнотекстовый поиск PostgreSQL (GIN/tsvector) вместо `contains` при объёме свыше 10 000 issues; cursor pagination, индексы `[teamId,statusId]`, `[teamId,updatedAt]`, `[assigneeId,statusId]`, `[projectId,updatedAt]`.
- Realtime подключать после корректной авторизации: обновления в пределах команды, reconciliation с локальным optimistic state, reconnect/backoff и индикатор offline.
- Хранить timestamps в UTC, timezone применять только при отображении и расчёте бизнес-календаря. Проверить перенос SQLite дат отдельным migration test.
- Добавить индексы для рабочих выборок: notifications `[userId, read, createdAt]`, comments/history `[issueId, createdAt]`, projects `[teamId, updatedAt]`.
- Определить SLO эксплуатации: RPO <= 15 минут, RTO <= 4 часов; ежеквартально проводить документированный restore drill из backup.
- Добавить `/health/live` без зависимости от БД и `/health/ready` с коротким `SELECT 1`; логи — структурированные, без session/token/password и содержимого приватных вложений.

## 8. Технические правила реализации

- TypeScript strict: не использовать `any`; создать DTO / Prisma select types и unit-тесты для mapper-ов.
- Server Actions не должны принимать «свободный» object. Принимать только валидированный DTO, ограниченный whitelist полей.
- Запросы данных из page-level server component; client components оставлять для интерактивности. Использовать React Query или эквивалент единообразно, если выбран client cache.
- URL и display text локализовать: язык интерфейса определяется настройкой пользователя; стартовая локаль `ru`, fallback `en`.
- Конфигурацию workflow, priority, SLA и feature flags хранить в БД/конфиге, а не жёстко в JSX.
- Добавить unit (валидация, ID generation, authorization), integration (Server Actions с тестовой БД) и E2E (Playwright) тесты.
- Добавить scripts: `typecheck`, `test`, `test:integration`, `test:e2e`, `db:validate`, `db:migrate`. Сейчас три `node:test` теста не подключены к `npm test` и проверяют в основном конфигурацию SQLite/деплоя.
- CI gate: `prisma validate` -> migration на пустую PostgreSQL БД -> migration test на копии SQLite fixture -> lint -> typecheck -> unit/integration -> build -> E2E smoke.
- Устранить build warnings: мигрировать deprecated `middleware.ts` в `proxy.ts` по поддерживаемому Next.js convention и явно зафиксировать workspace root/Turbopack root; не удалять внешний lockfile без согласования владельца.
- Для PostgreSQL поиска явно определить collation/locale и поведение регистра. Тесты обязаны покрывать ID, латиницу и кириллицу; не полагаться на SQLite `contains` semantics.
- Валидировать обязательные environment variables при старте приложения и документировать local/test/staging/production profiles в `.env.example`/runbook.

Минимальный E2E набор:

1. Login -> создать bug по шаблону -> прикрепить разрешённый файл -> увидеть задачу в списке.
2. Участник команды A не видит и не меняет issue команды B.
3. Перенести issue на доске -> optimistic state -> успешное сохранение и запись history.
4. Смоделировать ошибку мутации -> карточка откатывается, есть error toast.
5. Фильтры списка попадают в URL; ссылка восстанавливает тот же результат.
6. Keyboard-only: Cmd/Ctrl+K -> Create Issue -> заполнить -> сохранить.

## 10. Definition of Done для каждой фазы

- Все acceptance criteria фазы автоматизированы либо имеют воспроизводимый manual test case.
- `npm run lint` проходит без ошибок; `npm run build` проходит.
- Нет debug logging, mock data, `TODO` в поставляемом сценарии и незащищённых мутаций.
- Изменения схемы содержат миграцию, план отката и проверку существующих данных.
- `prisma db push` отсутствует в стандартных local/staging/production workflows; schema drift проверяется автоматически.
- PostgreSQL restore drill и migration report приложены к релизу P0.
- Выполнены desktop + mobile visual QA и базовая accessibility проверка.
- Документация: обновлены модель данных, permissions matrix, changelog и runbook миграции.

## 11. Release gate для P0

P0 нельзя считать завершённым, пока одновременно не выполнены все условия:

- local/test, staging и production используют PostgreSQL и одну цепочку Prisma migrations;
- migration report подтверждает перенос всех строк, attachments и business IDs; sequence и внешние ключи проверены;
- утверждена матрица `TeamMember`, а cross-team negative tests проходят для чтения и каждой мутации;
- публичная регистрация отключена или invitation-only, default passwords отсутствуют, приватные файлы не доступны по постоянному публичному URL;
- `prisma db push`, debug scripts/logs и mock data исключены из production workflow;
- `npm run lint`, typecheck, unit/integration, build и E2E smoke проходят в CI на PostgreSQL;
- выполнены staging cutover rehearsal, backup restore drill и документированная проверка rollback приложения;
- ручной smoke test подтверждает: login -> создать issue -> добавить comment/file -> изменить status/assignee -> получить notification -> найти issue;
- нет открытых Critical/High дефектов безопасности и потери данных; остальные известные ограничения внесены в release notes.

Входные данные, которые владелец продукта должен предоставить до начала cutover: выбранный PostgreSQL provider/region, утверждённая матрица участников команд, retention для вложений и audit log, maintenance window и ответственный за принятие migration report.

## 12. Явно вне текущего объёма

Не включать в первые две фазы: AI triage/автоназначение, публичный клиентский портал, сложную диаграмму Ганта, биллинг, marketplace плагинов и полную offline-first синхронизацию. Они оправданы только после того, как P0/P1 показатели использования и качества подтверждены.
